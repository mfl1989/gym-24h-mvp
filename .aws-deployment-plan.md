# ☁️ 24h 无人健身房 MVP - AWS 生产环境部署宪法 (.aws-deployment-plan.md)

## 0. 🚨 全局强制约束 (Global Rules)
- ❌ **严禁**在代码、配置文件、Dockerfile 中硬编码任何敏感凭证（密码、API Key、Secret）。
- ❌ **严禁**将 RDS 和 ECS Fargate 放置在 Public Subnet（必须绝对内网隔离）。
- ❌ **严禁** ECS Task 直接暴露公网 IP（必须通过 ALB 访问）。
- ❌ **严禁**跳过 HTTPS（所有公网流量必须通过 TLS 加密）。
- ❌ **严禁**将本地配置（`application-postgres-local.yml`）直接用于生产环境。
- ✅ 所有敏感配置必须通过系统环境变量（ENV）或 AWS SSM / Secrets Manager 注入。
- ✅ 所有后端服务必须将日志输出到标准输出（stdout），由 AWS CloudWatch 统一收集。

## 1. 架构选型与服务映射 (AWS Native Services)
本项目采用 Serverless + 托管服务优先策略，完美契合 AWS SAA 最佳实践：
- **前端接客区 (Vite SPA):** `Amazon S3` (静态托管) + `Amazon CloudFront` (CDN 加速与 HTTPS 卸载)。
- **核心业务层 (Spring Boot):** `Amazon ECR` (镜像仓库) + `Amazon ECS on Fargate` (无服务器容器) + `ALB` (应用负载均衡器)。
- **数据持久层 (PostgreSQL):** `Amazon RDS for PostgreSQL`（MVP 阶段采用 Single-AZ 单可用区部署以控制成本）。
- **网络基础设施:** 自定义 `VPC`，划分 Public Subnets (给 ALB 和 NAT) 与 Private Subnets (给 Fargate 和 RDS)。

## 2. 🔐 安全与网络强制规范 (Security & Networking)
### 2.1 网络与出口路由（🚨 核心生命线）
- Fargate 部署于 Private Subnet，绝对禁止分配公网 IP。
- **必须在 Public Subnet 部署 1 个 NAT Gateway**，并配置私有子网的路由表指向它。这是确保 Fargate 能够出网调用 Stripe、LINE 和 IoT 硬件等第三方公网 API 的**唯一安全通道**。
- *成本控制：* MVP 阶段仅需 1 个 NAT Gateway 即可，无需多可用区冗余。

### 2.2 安全组规则 (Security Groups) 必须严格遵守最小权限原则：
- **ALB 安全组:** 入站仅允许 `80 / 443`（来自 `0.0.0.0/0`）。
- **ECS Fargate 安全组:** 入站仅允许 `8080` 端口，且来源**必须指定为 ALB 的安全组 ID**（拒绝任何直连）。
- **RDS 安全组:** 入站仅允许 `5432` 端口，且来源**必须指定为 ECS Fargate 的安全组 ID**。

### 2.3 证书与 HTTPS
- 使用 **AWS Certificate Manager (ACM)** 免费签发 SSL 证书。
- 必须分别在 us-east-1（供 CloudFront 使用）和 ap-northeast-1（供 ALB 使用）申请证书。

## 3. 🗄️ 数据库规范 (Amazon RDS)
- **高可用与备份:** 必须启用自动备份（Retention ≥ 7天）。
- **应用侧保护:** Spring Boot 必须配置 HikariCP 连接池，限制最大连接数（如 `maximum-pool-size: 10`），防止并发洪峰打爆 RDS。

## 4. 🔑 生产环境变量抽取清单 (Environment Variables Matrix)
在生产环境的 ECS Task Definition 中，必须注入以下环境变量：

### 4.1 数据库组
| 变量名 | 说明 | 示例值 |
| :--- | :--- | :--- |
| `SPRING_DATASOURCE_URL` | RDS 实例的内网端点 | `jdbc:postgresql://<rds-endpoint>:5432/gym_db` |
| `SPRING_DATASOURCE_USERNAME` | DB 用户名 | `postgres` |
| `SPRING_DATASOURCE_PASSWORD` | DB 密码 | *(通过 ECS 挂载 Secrets Manager)* |

### 4.2 支付组 (Stripe)
| 变量名 | 说明 | 示例值 |
| :--- | :--- | :--- |
| `STRIPE_API_KEY` | Stripe 生产私钥 | `sk_live_...` |
| `STRIPE_WEBHOOK_SECRET` | 线上 Webhook 密钥 | `whsec_...` (从 Stripe 后台获取) |
| `STRIPE_PRICE_ID` | 真实包月商品 ID | `price_...` |

### 4.3 鉴权与前端跨域组 (LINE)
| 变量名 | 说明 | 示例值 |
| :--- | :--- | :--- |
| `LINE_CHANNEL_ID` | LINE 登录渠道 ID | 从 LINE Console 获取 |
| `CORS_ALLOWED_ORIGINS` | 允许跨域的前端域名 | `https://app.gym24h.jp` (CloudFront 域名) |

### 4.4 物理门禁组 (IoT)
| 变量名 | 说明 | 示例值 |
| :--- | :--- | :--- |
| `DOOR_LOCK_MOCK` | 硬件 Mock 开关 | **生产环境必须强制为 `false`** |
| `DOOR_LOCK_API_URL` | 真实硬件 API 地址 | 视服务商而定 |

## 5. 🧭 实施路径规划 (Execution Phases)
请 AI 助手在后续执行中，严格按照以下 Phase 顺序推进：

- **Phase A：容器化改造** (当前任务)
  编写生产级轻量化 `Dockerfile`，并创建纯粹依赖环境变量的 `application-prod.yml`。
- **Phase B：前端云准备**
  配置 Vite 环境变量，注入未来的 ALB 线上域名，确保前端可以被 build 为纯静态资源。
- **Phase C：基础设施搭建 (AWS Console 手工操作阶段)**
  创建 VPC、Public/Private Subnet、NAT Gateway、RDS、ALB。
- **Phase D：MVP 首次上线 (Day 1)**
  本地打镜像并推送到 ECR，在 ECS 控制台手动创建 Fargate Task Definition 挂载环境变量并启动，配置 ALB 路由，验证全链路。
- **Phase E：生产环境进阶 (Post-MVP 加固 - 赚到钱后再做)**
  配置 GitHub Actions 实现自动化 CI/CD，在 ALB 前方接入 AWS WAF 阻挡恶意攻击，为 ECS 配置基于 CPU/Mem 的 Auto Scaling。
